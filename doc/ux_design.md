# Atoms 用户体验设计

> 本文档描述用户从零开始使用 Atoms 的完整旅程、界面结构、核心交互与设计原则。
> 基于《原子习惯》方法论，但**不假设用户读过书**。

---

## 一、核心理念

### 用户来的时候带的是目标，不是身份

```
目标 → 里程碑 → 行动计划 → 习惯（行动项容器） → 回顾 → 身份浮现
                   ↑                                              │
                   └────────── 身份反哺系统设计 ←─────────────────┘
```

**关键层次区分：**
- **行动计划**：为了达成这个里程碑，需要做哪些事？（思考层：罗列行为）
- **习惯**：把这些事按频率组织成可执行的训练单元（组织层：打包+排期）

- **目标**：用户被结果吸引而来，"想减10公斤"、"想完成双力臂"
- **里程碑**：拆解为有进度的阶段性关口
- **习惯**：服务于里程碑的执行单元（有频率、有行动项清单）
- **身份**：不是用户填的表单，是积累足够行动证据后系统返回的洞察

### 一体两面

| 面 | 用途 | 时机 |
|------|------|------|
| **习惯系统面** | 今天该做什么？逐项打勾执行 | 每天使用，默认主页 |
| **目标面** | 离目标还有多远？系统哪里卡了？ | 需要确认进度时 |

两面通过左右滑动切换，对等关系，非主从关系。

## 二、用户旅程

### 阶段 0：首次使用 — 接住动机（~3分钟）

用户第一次打开应用，不看到表格、不看到术语、不看到空列表。

#### 卡片 1：接住目标

```

│        你想达成什么？                │
│                                     │

│    │                         │      │
│    └─────────────────────────┘      │
│                                     │
│    比如：                           │
│    · 减掉 10 公斤                   │
│    · 完成一个双力臂                 │

│    · 每天醒来精力充沛               │
│                                     │
│                    [ 下一步 ]        │
└─────────────────────────────────────┘
```

#### 卡片 2：拆里程碑

```
┌─────────────────────────────────────┐
│                                     │
│    要达成"完成双力臂"，              │
│    路上需要过哪些关口？             │
│                                     │
│    ┌─────────────────────────┐      │
│    │ ① 完成 1 个引体向上     │      │
│    └─────────────────────────┘      │
│    ┌─────────────────────────┐      │
```

- 提供模板建议：如果用户不知道拆什么（如健身类模板、学习类模板）
- 最少 1 个里程碑即可继续
- 里程碑可以用一个短句描述，再加一个可量化的判定条件（自动或手动）

```
│    └──────────────────────────┘      │
│    ┌──────────────────────────┐      │
│    │ · 离心引体下降 5×3        │      │
│    └──────────────────────────┘      │
│    ┌──────────────────────────┐      │
│    │ · 拉伸 30秒               │      │
│    └──────────────────────────┘      │
│    ┌──────────────────────────┐      │
│    │ · 平板支撑 60秒           │      │
│    └──────────────────────────┘      │
│    ┌──────────────────────────┐      │
│    │ · 死虫式 3×10             │      │
```

- 这一步的要点：**只管列，不管排。** 把所有想到的行为都写下来。
- 每个行动是一条文本，无格式要求
- 后面设计习惯时可以引用这个清单，不需要重新输入

```
    │    │                            │    │

    │    │ ☐ 死虫式 3×10             │    │
    │    │                            │    │
    │    │ 如果太累不想做完整版，     │    │
    │    │ 两分钟能做什么？           │    │
```

- 行动计划自动出现在候选清单中，勾选即可
- 可以建多个习惯，同一个行动项可以被不同习惯复用（如拉伸）
- 如果还剩未分配的行动，完成后提示"还有 X 个行动没分配，确定完成？"
- 后续添加习惯时，也会展示当前里程碑的行动计划供引用
- **两分钟安全阀**：用户自己定义"最低限度版本"。不是偷懒许可，而是防止彻底不做的兜底策略。输入提示："如果太累，两分钟你能做什么？"

### 阶段 1：日常执行 — 习惯系统面（默认主页）

这是用户每天打开看到的主界面。

```
┌──────────────────────────────────────┐
│  Atoms                    ○ 回顾 │   │
├──────────────────────────────────────┤
│                                      │
│  当前里程：完成 1 个引体向上          │
│  ████░░░░░░  已完成 3 次 / 目标达标   │
│                                      │
│  ┌────────────────────────────────┐  │
│  │ 练背计划        每两天一次      │  │
│  │                                │  │
│  │ ☑ 负重悬吊 30秒                │  │
│  │ ☑ 弹力带辅助引体 5×3           │  │
│  │ ☐ 离心引体下降 5×3             │  │
│  │ ☐ 拉伸 30秒                    │  │
│  │                                │  │
│  │      [ 全部完成 ] [ 只做两分钟 ]│  │
│  │      [ 跳过今天 ]              │  │
│  └────────────────────────────────┘  │
│                                      │
│  ┌────────────────────────────────┐  │
│  │ 核心训练        每周两次        │  │
│  │                                │  │
│  │ ☐ 平板支撑 60秒                │  │
│  │ ☐ 死虫式 3×10                  │  │
│  └────────────────────────────────┘  │
│                                      │
│         [ + 添加习惯 ]               │
│                                      │
│    ←─ 目标面  │  习惯面  ─→         │
└──────────────────────────────────────┘
```

**交互规则：**

| 操作 | 行为 |
|------|------|
| 点击行动项 | 打勾/取消（当天不可取消，需确认） |
| 点击"全部完成" | 全部行动项打勾，状态记为 `full`，弹出确认提示 |
| 点击"只做两分钟" | 卡片折叠，只展示两分钟安全阀文本 + 一键确认按钮。确认后状态记为 `two_min` |
| 点击"跳过今天" | 全部跳过，状态记为 `skipped`。不触发"绝不连续错过两次"警告（第一天没事） |
| 某行动项部分完成 | 可部分提交，状态记为 `two_min` |
| 两分钟模式下的升级 | 做完两分钟后，弹窗提示"要继续完成完整版吗？"，可选升级为 `full` |
| 长按习惯卡片 | 进入编辑模式，调整行动项、两分钟安全阀文本或删除 |
| 左滑 | 切换到目标面 |
| 顶部"回顾"按钮 | 进入本周回顾 |

**里程碑状态与判定：**

| 状态 | 含义 | 判定方式 |
|------|------|---------|
| 等待 | 尚未激活 | 上一个里程碑完成后自动激活 |
| 进行中 | 当前正在攻克 | 用户激活或自动激活 |
| 已完成 | 达标 | 用户手动确认 或 自动判定（可量化时） |

- 里程碑进度可以由用户手动更新（如"引体从 3 个做到了 7 个"）
- 也可以关联到习惯数据，自动估算

## 三、核心设计原则

### 1. 语言原则

| 避免 | 改用 |
|------|------|
| 身份声明 | 不出现，直到系统浮现 |
| 两分钟版本 | "如果只想做两分钟"（内联提示） |
| 叠加线索 | "在什么之后做这件事？"（对话式） |
| 习惯名称 | 用户自己起，不需要术语 |

### 2. 空状态原则

每个空状态回答两个问题：**这是什么** 和 **为什么要做**。

```
没有习惯 ← 不要这样
↓
"为'完成 1 个引体'设计一个训练计划吧
 把你要做的动作列出来，每次打卡推进进度。"
```

### 3. 身份原则

- 身份不是用户创建的数据
- 身份是系统基于行动证据生成的洞察
- 用户可以选择认可、修改或忽略

### 4. 目标面与习惯面的关系

- 两面对等，左右滑动切换
- 从目标面发现系统问题时，可点击跳转到对应习惯
- 从习惯面完成打卡后，可滑动查看进度更新

### 5. 行动计划与习惯的设计原则

- **行动计划是罗列层**：列出"为了达成这个里程碑需要做什么"，先发散，后筛选
- **习惯是组织层**：将行动计划按频率打包成可执行的训练单元
- 卡片 3（行动计划）和卡片 4（设计习惯）是连续两步，不合并
- 设计习惯时，行动计划自动作为候选清单，勾选即可，无需重新输入
- 后续新增习惯时，同样参考已有的行动计划

### 6. 两分钟安全阀

- 不是习惯的另一个版本，是同一个习惯的**兜底策略**
- 用户自己定义："如果太累，两分钟能做什么？"
- 默认目标永远是"全部完成"，两分钟是安全阀
- 做完两分钟后提示升级，保留"做了总比没做好"的累积价值
- 两分钟完成算一次执行，计入日志，推进进度

### 7. 里程碑原则

- 里程碑是**可验证的阶段性成果**
- 只有当前里程碑激活其习惯
- 里程碑完成 → 旧习惯可选择归档 → 新里程碑激活 → 设计新习惯

## 四、数据模型

```
goals                               # 目标
  id, name, status, created

milestones                          # 里程碑
  id, goal_id, name, order, status,
  target_desc, current_value, target_value,
  created, completed_at

action_plans                        # 行动计划（里程碑的原始行为清单）
  id, milestone_id, name, sort,
  created

habits                              # 习惯（将行动计划按频率组织的执行单元容器）
  id, milestone_id, name, frequency,
  two_min_ver,                      # 两分钟安全阀文本（用户自定义）
  archived, created

habit_actions                       # 习惯-行动关联（多对多）
  id, habit_id, action_plan_id, sort

logs                                # 执行日志
  id, habit_id, date, status,       # two_min | full | skipped
  action_completions,               # JSON: {action_plan_id: bool, ...}
  note, created

reviews                             # 每周回顾（保持现有结构）
  id, week, notes, created
```

**数据关系说明：**

- `action_plans` 属于里程碑，列出"需要做什么"（罗列层）
- `habits` 属于里程碑，定义"怎么安排执行"（组织层）
- `habit_actions` 是多对多关联：一个行动可被多个习惯引用（如拉伸），一个习惯包含多个行动
- 创建习惯时，从 `action_plans` 勾选，自动写入 `habit_actions`

`identities` 表改为由系统生成，不是用户创建：
- 系统基于 `logs` 数据分析浮现
- 存储为洞察记录，用户可选择是否认可
